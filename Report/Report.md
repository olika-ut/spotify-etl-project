# Spotify ETL and Data Visualization Project Report

Group: Olga Khodyreva, Aleksei Safjanov, Jevgeni Filatov

The aim of this project was to build a complete ETL pipeline and create visualizations that tell a clear story with the data. The project focused on understanding the relationship between Spotify track popularity and music genres. The data came from the Spotify Tracks Dataset on Kaggle. The main objective was to explore how track popularity differs across genres and whether audio features such as danceability, energy, valence, and acousticness are connected to popularity. The research question was: How do genre and audio characteristics relate to Spotify track popularity?

At the beginning of the project, an important step was selecting the dataset. Different options were considered, including Spotify datasets from Kaggle and public databases such as MusicBrainz. MusicBrainz appeared to be more reliable because it is an official public music database. However, it was also more difficult to use because it contains many connected tables and does not directly provide Spotify-style audio features such as popularity, danceability, energy, and valence. For this reason, the Kaggle Spotify Tracks Dataset was selected. It was easier to use for an ETL and visualization project and included useful variables for analysis. Since Kaggle datasets are user-contributed, the results of this project should be considered exploratory rather than official Spotify statistics.

The ETL process was completed in R using RStudio. In the extract stage, the raw CSV dataset was imported and examined. The dataset contained around 114,000 rows and 21 columns. The first script checked the column names, previewed the first records, and identified missing values. During the transform stage, the data was cleaned and prepared for analysis. An unnecessary index column was removed, rows with missing important values were deleted, duplicate tracks were removed based on track_id, and track duration was converted from milliseconds into minutes. In addition, a new column called popularity_category was created to group tracks into low, medium, and high popularity levels. A genre summary table was also generated to calculate average popularity and average audio features for each genre. In the load stage, the cleaned dataset and summary table were stored in a SQLite database. This improved the organization of the project and made the pipeline more reproducible.

The visualization stage was first completed in R using the ggplot2 package. Several charts were created to analyze the data from different perspectives. The first chart displayed the top 15 genres by average popularity. The second chart compared genres by average energy and valence, while popularity was represented through point size and color. Another visualization showed average popularity according to low, medium, and high levels of audio features. A stacked bar chart illustrated the distribution of popularity categories within the top genres, and a correlation heatmap highlighted relationships between numerical variables. Overall, the visualizations showed that k-pop, pop-film, and metal had the highest average popularity scores in the dataset. At the same time, the results suggested that popularity cannot be explained by a single audio feature. Tracks with medium levels of danceability, energy, valence, and acousticness often achieved slightly higher popularity scores.

An important part of the project involved learning and experimenting with different tools. After creating visualizations in RStudio, Tableau was also tested because it supports interactive dashboards and offers more flexible design options. Tableau was useful for building an interactive data story and experimenting with colors, labels, filters, and dashboard layouts. A green color palette was selected to reflect the Spotify theme. However, creating effective visualizations was not only about using software tools. One challenge was deciding which charts were truly meaningful and supported the overall story. Some charts looked visually attractive but did not contribute much to the analysis, so the final visualizations had to be chosen carefully.

Several practical challenges also appeared during the project. The raw dataset was too large to upload to GitHub, so it was excluded from the repository, and the README file explains that users must download it separately from Kaggle. Some Tableau visualizations also required additional summary CSV files, which meant that extra processed files had to be exported from R. Git and GitHub created additional difficulties, especially when connecting the local project folder to the online repository and solving merge conflicts. These issues were resolved step by step by using Git commands, updating the README file, and uploading the final project files to GitHub.

Overall, this project provided valuable practical experience with the complete data workflow: selecting a dataset, cleaning and transforming data, loading it into a database, creating visualizations, experimenting with Tableau dashboards, documenting the pipeline, and using GitHub for version control. Although the dataset itself was not highly complex, the project demonstrated how many decisions are involved in creating a clear, organized, and reproducible data analysis project.
